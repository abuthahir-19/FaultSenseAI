1. Hybrid search (dense vector + BM25)

┌──────────────┬───────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│              │                                                                                                               │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ What we use  │ ChromaDB semantic search + in-memory BM25, fused with Reciprocal Rank Fusion (RRF) (hybrid_retriever.py, RRF_ │
│              │ K=60).                                                                                                        │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Alternatives │ • Semantic-only (embeddings only)                                                                             │
│              │ • Keyword-only (Elasticsearch/Solr BM25)                                                                      │
│              │ • Native hybrid DB (Weaviate, Vespa, Elasticsearch dense_vector + BM25)                                       │
│              │ • LangChain EnsembleRetriever (weighted merge of retrievers)                                                  │
│              │ • Learned rerankers only (cross-encoder, no fusion)                                                           │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Why chosen   │ Telecom queries mix paraphrases (“coverage degradation”) and exact tokens (“Ericsson RRU”, alarm IDs).        │
│              │ Semantic-only misses the latter; BM25-only misses the former. RRF merges ranked lists without normalizing     │
│              │ incompatible scores (Cormack et al., k=60). Documented gain: ~23% better top-5 recall vs semantic-only on     │
│              │ this dataset. Custom RRF vs EnsembleRetriever: same idea, full control and no extra LangChain retrieval       │
│              │ abstraction.                                                                                                  │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Say in Q&A   │ “We treat hybrid search as a retrieval engineering problem, not a single-model problem—RRF is simple,         │
│              │ reproducible, and works well at our scale.”                                                                   │
└──────────────┴───────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

┌──────────────┬───────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│              │                                                                                                               │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ What we use  │ Each CSV row → one embedded document (description + resolution_notes); rich fields in Chroma metadata for     │
│              │ filtering.                                                                                                    │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

│              │ • Embed description only; metadata as filters only                                                            │

│ Why chosen   │ Alarms are already semi-structured; splitting would break the link between symptom and resolution.            │
│              │ Concatenating description + resolution keeps RCA and fix steps in the same vector. Metadata duplicated for    │
│              │ hard pre-filter before vector search.                                                                         │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Say in Q&A   │ “We chunk at the business unit—the incident record—not arbitrary token windows.”                              │

7. LLM reranking (/api/rerank)

┌──────────────┬───────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│              │                                                                                                               │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ What we use  │ LLM scores each incident 0–1; combined_score = 0.6 × judge + 0.4 × RRF; re-sort.                              │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Alternatives │ Pure cross-encoder (e.g. ms-marco-MiniLM); pure LLM judge; Cohere Rerank API; no rerank                       │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Why chosen   │ Pure LLM-per-doc is accurate but slow/expensive. Blend keeps most of the gain with fewer calls and still      │
│              │ fixes “high BM25 but wrong fault” cases.                                                                      │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Say in Q&A   │ “Reranking is a precision layer on top of cheap hybrid retrieval.”                                            │
└──────────────┴───────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

10. Multi-agent troubleshooting — LangGraph

┌──────────────┬───────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│              │                                                                                                               │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ What we use  │ StateGraph on FaultAnalysisState: retrieval → correlation → RCA → service impact (branch) → resolution.       │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Alternatives │ CrewAI (role chat), AutoGen (conversation), single monolithic prompt, Supervisor pattern (dynamic routing),   │
│              │ sequential Python functions                                                                                   │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Why chosen   │ Pipeline is mostly fixed with one branch (escalation)—LangGraph gives typed state, conditional edges,         │
│              │ reasoning_trace per step for the UI, and unit-testable nodes. CrewAI/AutoGen: flexible but harder to audit    │
│              │ for regulated NOC workflows. No separate “supervisor” agent: graph topology is the supervisor.                │
├──────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Say in Q&A   │ “LangGraph gives us a state machine NOC engineers can follow, not a black-box chat.”                          │
└──────────────┴───────────────────────────────────────────────────────────────────────────────────────────────────────────────┘
